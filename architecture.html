Conceito 1

Clean Architecture

Criada por Robert C. Martin ("Uncle Bob"), é uma forma de organizar o software em camadas concêntricas, onde o código de alto nível (regras de negócio) não depende de detalhes de infraestrutura como banco de dados, UI ou frameworks externos.

Frameworks & Drivers UI, HTTP, DB, Flutter Widgets
Interface Adapters ViewModels, Repositories, Controllers
Use Cases Regras de aplicação
Entities Regras de negócio

🔑 A Regra de Dependência

As dependências do código-fonte devem apontar somente para dentro. A camada externa conhece a interna, mas jamais o contrário.

⚡ O Domain define contratos (interfaces). O Data os implementa. O Domain nunca importa o Data.

Domain

Núcleo da aplicação. Contém entidades e regras de negócio puras. Zero dependência de frameworks ou infraestrutura.

📄 game_model.dart 📄 ai_repository.dart 📄 game_rule.dart 📄 game_victory_condition.dart
Data

Implementa os contratos do domain. Faz a ponte entre o domain e fontes de dados externas como APIs ou banco.

📄 ai_repository_impl.dart 📄 ollama_service.dart
Presentation

Camada de UI e ViewModels. Consome o domain via abstrações. Nunca acessa datasources diretamente.

📄 home_view_model.dart 📄 app.dart 📁 ui/home/
Core / Infra

Utilitários transversais reutilizáveis por todas as camadas: cliente HTTP, extensões, configurações.

📄 network_client.dart 📄 network_interface.dart
Conceito 2

Princípios SOLID

Cinco princípios de design orientado a objetos que tornam o código mais manutenível, extensível e testável. Cada letra da sigla representa um princípio.

S
Single Responsibility Principle
Uma classe deve ter apenas um motivo para mudar. Cada classe é responsável por uma única parte da funcionalidade.
⚠️ No projeto: GameRule mistura lógica de negócio (verificar vencedor) com orquestração de fluxo (chamar AI, gerenciar estado). Deveria ser dividida em um Use Case e uma classe de regras puras.
O
Open/Closed Principle
Entidades de software devem estar abertas para extensão, mas fechadas para modificação. Extend, não altere.
✅ No projeto: NetworkClient é uma classe abstrata que pode ser estendida por novos serviços (como OllamaService) sem ser modificada. Bem aplicado.
L
Liskov Substitution Principle
Subclasses devem poder substituir suas superclasses sem alterar o comportamento esperado do programa.
✅ No projeto: AiRepositoryImpl implements AiRepository. Qualquer lugar que usa AiRepository pode receber AiRepositoryImpl sem quebrar comportamento.
I
Interface Segregation Principle
Clientes não devem ser forçados a depender de interfaces que não utilizam. Prefira interfaces pequenas e específicas.
✅ No projeto: NetworkInterface e AiRepository são interfaces mínimas e coesas, com apenas os métodos necessários. Bem aplicado.
D
Dependency Inversion Principle
Módulos de alto nível não devem depender de módulos de baixo nível. Ambos devem depender de abstrações. Abstrações não devem depender de detalhes — detalhes dependem de abstrações. 
// ❌ VIOLAÇÃO no GameRule (domain)
final AiRepository _aiRepo =
  DependencyInjection()
    .aiRepositoryImpl; // busca concreta!
❌ Domain acessa o service locator e busca a implementação concreta. Isso inverte a inversão de dependência. 
// ✅ CORRETO — injeção via construtor
class GameRule {
  final AiRepository _aiRepo;

  GameRule({
    required AiRepository repo
  }) : _aiRepo = repo;
}

Problemas & Pontos Positivos

Análise detalhada de cada aspecto do projeto em relação às premissas arquiteturais.

✅
Separação de camadas domain / data / presentation
A estrutura de pastas respeita as fronteiras da Clean Architecture. O domain define contratos, o data implementa, e a presentation consome via abstração.
✅
AiRepository como abstração no domain
AiRepository (abstract) vive no domain e AiRepositoryImpl no data. O fluxo de inversão de dependência está correto nessa relação.
✅
GameModel como entidade pura
GameModel não contém imports de Flutter, MobX ou qualquer framework. É uma classe Dart pura — exatamente como deve ser no domain.
❌
GameRule acessa DependencyInjection internamente DIP
GameRule (domain) instancia DependencyInjection().aiRepositoryImpl direto no campo. Isso viola tanto o DIP quanto a Dependency Rule da Clean Architecture — o domain não pode depender da camada de injeção. 
final AiRepository _aiRepo =
  DependencyInjection().aiRepositoryImpl; // ❌ domain → DI
final AiRepository _aiRepo;           // ✅ receba via construtor
❌
HomeViewModel acessa DependencyInjection internamente DIP
Mesmo problema: HomeViewModelState busca DependencyInjection().gameRule ao invés de receber a dependência injetada. Isso torna o ViewModel impossível de testar de forma isolada.
❌
DependencyInjection expõe implementações concretas DIP
late final AiRepositoryImpl aiRepositoryImpl deveria ser late final AiRepository aiRepository. Expor concretas força todos os consumidores a conhecer os detalhes de implementação.
❌
OllamaService duplicado — código morto em lib/service/ SRP
Existe uma cópia idêntica do OllamaService em lib/service/ollama/ que nunca é importada. A pasta service/ não tem papel definido na arquitetura e deve ser removida.
⚠️
Variáveis globais mutáveis em game_victory_condition.dart SRP
List<int> verticalFirstColumn = [0,3,6] são variáveis globais que podem ser mutadas acidentalmente. Deveriam ser constantes encapsuladas em uma classe.
⚠️
Bug lógico: diagonalFirstCondition incorreto
diagonalFirstCondition = [1,4,8] está errado. A diagonal principal do Tic-Tac-Toe é [0,4,8]. Índice 1 é a coluna do meio na primeira linha, não um canto.
⚠️
Ausência de Use Cases explícitos
A Clean Architecture recomenda uma camada de Use Cases separada. Atualmente, GameRule acumula responsabilidades que seriam melhor divididas em PlayMoveUseCase, ResetGameUseCase, etc.
